## Passo 1 — #9: `downgrade()` da migration `6350f4a6ba31` inexecutável

**Achado:** `op.create_unique_constraint(None, 'usuario', ['convite_token'])`
no `upgrade()` e `op.drop_constraint(None, 'usuario', type_='unique')` no
`downgrade()` usam `None` como nome de constraint — Alembic exige um nome
real pra `drop_constraint`; rodar o downgrade falha.

**Arquivo:** `services/prisma-api/alembic/versions/6350f4a6ba31_cadastro_status_cadastro_convite_token_.py`

**Mudança:**
```python
def upgrade() -> None:
    ...
    op.create_unique_constraint("uq_usuario_convite_token", "usuario", ["convite_token"])
    # ### end Alembic commands ###


def downgrade() -> None:
    # ### commands auto generated by Alembic - please adjust! ###
    op.drop_constraint("uq_usuario_convite_token", "usuario", type_="unique")
    op.drop_column('usuario', 'convite_expira_em')
    op.drop_column('usuario', 'convite_token')
    op.drop_column('usuario', 'status_cadastro')
    sa.Enum(name='status_cadastro_enum').drop(op.get_bind(), checkfirst=True)
    # ### end Alembic commands ###
```

**Cuidado:** esta migration já rodou no banco de dev/produção (a coluna
`status_cadastro` já existe e está em uso — Passo 2 depende dela). Trocar o
nome da constraint no `upgrade()` já aplicado NÃO precisa de re-migração —
o nome de uma constraint existente no Postgres não muda sozinho só porque
o arquivo Python mudou. Se quiser verificar o nome real que o Postgres deu
à constraint existente (criada sem nome explícito, ganhou um nome
autogerado tipo `usuario_convite_token_key`), rode:
```sql
SELECT conname FROM pg_constraint WHERE conrelid = 'usuario'::regclass AND contype = 'u';
```
Se o nome autogerado for diferente de `uq_usuario_convite_token`, ajuste o
`downgrade()` pra usar o nome REAL que já existe no banco (não o que você
gostaria que fosse) — o `upgrade()` editado só vale pra quem rodar a
migration do zero num banco novo.

**Verificação:** `cd services/prisma-api && ../../.venv/bin/python -m alembic upgrade head && ../../.venv/bin/python -m alembic downgrade -1 && ../../.venv/bin/python -m alembic upgrade head` contra o banco `prisma_test` (não o de dev/produção!) deve completar sem erro. ## Passo 2 — #5 + #6: convite expirado sem reemissão + rejeitado permanentemente bloqueado

**Achados:**
- #5: não existe rota pra gerar um novo token quando o de um cadastro já
  `APROVADO` expira (só 48h de validade, `convite.TOKEN_EXPIRA_HORAS`).
- #6: `rejeitar_cadastro_rota` (app.py:1178-1198) muda `status_cadastro` pra
  `REJEITADO` e não existe nenhum caminho de volta — uma nova tentativa de
  `/auth/cadastro` com a mesma matrícula sempre esbarra no `409` de
  matrícula duplicada, porque a matrícula é única e permanece ligada ao
  registro rejeitado pra sempre.

**Decisão de design:** em vez de duas rotas novas separadas, resolva os
dois reaproveitando `aprovar_cadastro_rota` — ela já faz tudo que #5
precisa (gerar token novo + expiração + reenviar e-mail + devolver o
link), só precisa aceitar mais um status de origem.

**Mudança em `aprovar_cadastro_rota`** (`services/prisma-api/app.py`,
função inteira em torno da linha 1133-1175):

Troque:
```python
    if alvo.status_cadastro != _StatusCadastro.PENDENTE:
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail="cadastro não está pendente")
```
por:
```python
    if alvo.status_cadastro not in (_StatusCadastro.PENDENTE, _StatusCadastro.REJEITADO):
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST,
                            detail="cadastro precisa estar pendente ou rejeitado")
```

Isso já resolve #6 (rejeitado pode ser aprovado depois, reabrindo o
caminho). Para #5 (reemitir token de um cadastro JÁ aprovado, mas com
token expirado — não é o mesmo caso de #6, que é sobre reverter uma
rejeição), adicione uma nova rota logo depois de `aprovar_cadastro_rota`:

```python
@app.post("/usuarios/{usuario_id}/reenviar-convite", response_model=AprovarResp,
          dependencies=[Depends(auth.exigir_papel("gestor", "compliance")), Depends(auth.verificar_csrf)])
def reenviar_convite_rota(usuario_id: int, usuario: auth.UsuarioAtual = Depends(auth.get_usuario_atual),
                          db=Depends(get_db)):
    """Gera um novo token quando o anterior expirou (48h) e o usuário nunca
    ativou a conta — mesmo efeito de `aprovar_cadastro_rota`, mas pra quem
    já está `APROVADO` (não passa por Pendente de novo)."""
    from datetime import timedelta, timezone as _timezone

    import convite as _convite

    from db.models import StatusCadastro as _StatusCadastro
    from db.repo import buscar_usuario_por_id

    alvo = buscar_usuario_por_id(db, usuario_id)
    if alvo is None:
        raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail="usuário não encontrado")
    if alvo.gestora_id != usuario.gestora_id:
        raise HTTPException(status_code=status.HTTP_403_FORBIDDEN, detail="usuário de outra gestora")
    if alvo.status_cadastro != _StatusCadastro.APROVADO:
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail="cadastro não está aprovado")
    if alvo.ativo and alvo.senha_hash and not alvo.convite_token:
        # já ativou a conta (tem senha própria, sem convite pendente) — reenviar não faz sentido
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST, detail="conta já foi ativada")

    token = _convite.gerar_token()
    alvo.convite_token = token
    alvo.convite_expira_em = datetime.now(_timezone.utc) + timedelta(hours=_convite.TOKEN_EXPIRA_HORAS)
    db.commit()

    link = _link_ativacao(token)
    email_enviado = _convite.enviar_email_ativacao(alvo.email, alvo.nome, link) if alvo.email else False

    audit.registrar_evento(rota="/usuarios/reenviar-convite", ator_matricula=usuario.matricula,
                           descricao=f"convite reemitido: {alvo.matricula}")
    return AprovarResp(link_ativacao=link, email_enviado=email_enviado)
```

**Nota sobre "já ativou a conta":** o jeito mais simples de saber se uma
conta convidada já foi ativada é `alvo.convite_token is None` DEPOIS que
`ativar_conta_rota` limpa o token (confirme lendo essa rota — ela já limpa
`convite_token`/`convite_expira_em` ao ativar, igual ao comentário do
plano original de cadastro). Se essa premissa não bater ao ler o código
(ex.: o campo não é limpo na ativação), simplifique a condição pra só
`if alvo.convite_token is None: raise ... "conta já foi ativada"` e ajuste
o teste de acordo — não invente um campo novo pra isso.

**Verificação:** teste manual + os testes do Passo 6 abaixo cobrem isso.

## Passo 8 — #16: `POST /auth/2fa/confirmar` sem rate limit

**Achado:** `verificar_2fa` (rota irmã, login) tem rate limit explícito
com o comentário "Um TOTP de 6 dígitos é alvo de força bruta se não
limitado" — `confirmar_2fa` faz a mesma verificação de TOTP mas não tem
limitador. Exige sessão autenticada (menor severidade que `verificar_2fa`,
que é pré-login), mas é a mesma classe de vulnerabilidade já reconhecida
no código vizinho.

**Mudança em `app.py`** (~linha 444-461), seguindo o padrão exato de
`_login_com_rate_limit`/`_verificar_2fa_com_rate_limit`:
```python
@app.post("/auth/2fa/confirmar", dependencies=[Depends(auth.verificar_csrf)])
def confirmar_2fa(request: Request, req: Confirmar2FAReq,
                  usuario: auth.UsuarioAtual = Depends(auth.exigir_papel("gestor", "compliance")),
                  db=Depends(get_db)):
    """Fininho, mesmo motivo das outras rotas com rate limit (ver skill
    fastapi-slowapi-future-annotations)."""
    return _confirmar_2fa_com_rate_limit(request, req, usuario, db)


@limiter.limit("5/minute")
def _confirmar_2fa_com_rate_limit(request: Request, req: Confirmar2FAReq,
                                   usuario: auth.UsuarioAtual, db):
    import pyotp

    from db.models import Usuario as _Usuario
    alvo = db.get(_Usuario, usuario.id)
    if not alvo.totp_secret_pendente:
        raise HTTPException(status_code=status.HTTP_400_BAD_REQUEST,
                            detail="nenhum enrollment de 2FA em andamento — chame /auth/2fa/iniciar primeiro")
    if not pyotp.TOTP(alvo.totp_secret_pendente).verify(req.codigo, valid_window=1):
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED, detail="código inválido")
    alvo.totp_secret = alvo.totp_secret_pendente
    alvo.totp_secret_pendente = None
    alvo.totp_ativado = True
    db.commit()
    audit.registrar_evento(rota="/auth/2fa/confirmar", ator_matricula=usuario.matricula,
                           descricao="2FA ativado")
    return {"ok": True}
```

## Achados considerados e não incluídos

- **#7 (`PATCH /usuarios/{id}` não conhece `status_cadastro`):**
  resolvido *por design* pelo Passo 2, não por código novo em `PATCH`. Os
  endpoints de ação dedicada (`aprovar`/`rejeitar`/`reenviar-convite`) são
  o jeito certo de mudar `status_cadastro` — cada um valida a transição
  específica que faz sentido (ex.: só pode aprovar quem está
  pendente/rejeitado). Expor `status_cadastro` como campo solto de um
  `PATCH` genérico permitiria transições inválidas (ex.: pular de
  `PENDENTE` direto pra `APROVADO` sem gerar token, ou setar um valor
  fora do enum) sem a validação que os endpoints dedicados já fazem — API
  de ação explícita é mais segura que mutação de campo genérico aqui.
  Não crie uma rota de `PATCH` que aceite esse campo.

- **#8 (`409` em `/auth/cadastro` permite enumeração de matrícula):**
  achado real do ponto de vista do OWASP Account Enumeration Cheat Sheet,
  mas **decisão: manter o comportamento atual**. Razões: (1) o teste
  `test_autocadastro_matricula_duplicada_retorna_409` já testa esse
  contrato deliberadamente — mudar pra sempre-201 é uma mudança de
  comportamento de API observável, não um bug incidental; (2) matrícula
  de funcionário não é um segredo no mesmo nível de e-mail/senha — é
  frequentemente sequencial ou já conhecida dentro da própria
  organização; (3) a rota já tem rate limit (`5/minute`), o que já limita
  o volume de enumeração possível. Se o produto crescer pra fora do
  contexto interno de uma instituição (matrícula vira mais sensível),
  revisite esta decisão — mas não mude agora sem confirmar com o
  usuário, porque é uma troca de UX (o candidato hoje sabe na hora que
  digitou a matrícula errada ou já tem cadastro) por uma redução pequena
  de risco.

- **#15 (`matricula` é `unique` globalmente, não por `gestora_id`):** o
  próprio relatório da auditoria já sinalizou isso como possivelmente
  intencional (confiança "média", não "alta"). Mudar isso é uma decisão
  de modelo de negócio (matrícula é identificador por-empresa ou
  identificador nacional único?) que muda o schema (troca de
  `unique=True` solto pra `UniqueConstraint("gestora_id", "matricula")`)
  e precisa de nova migration. **Não decida isso sozinho** — está fora do
  escopo deste plano; volte pro usuário com a pergunta direta antes de
  tocar nisso.
